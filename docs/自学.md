## 泛型

T,E,K,V表示任意数据类型，等待装填，`<T>`表示在编译阶段检查并保证输出T类型的数据，只能是引用类型，不能是基本数据类型  
如List`<String>`,就是说这个类是列表类，但是泛型可以保证里面的成员都是String类型，否则编译不通过，如ResultUtil`<T>`这种，也是可以用泛型来规定数据的类型  

## 迭代器

场景二：遍历的时候删东西，因为普通for和加强for删的时候不管位置变动，迭代器可以优雅的边遍历边删除元素
  // 错误写法
  List`<String>` list = new ArrayList<>();
  list.add("A");
  list.add("B");
  list.add("C");
  for (String s : list) {
      if (s.equals("B")) {
          list.remove(s);  // 炸！ConcurrentModificationException
      }
  }
  // 正确写法：用迭代器
  Iterator`<String>` it = list.iterator();
  while (it.hasNext()) {
      if (it.next().equals("B")) {
          it.remove();  // 安全删除
      }
  }

  正常情况下怎么操作对象？

  user.setNickname("张三");   // 你知道有 setNickname 这个方法，直接调用

  反射干了什么：不知道也给你查出来

  // 给你一个不知道是什么的类
  Class<?> clazz = DpUser.class;

  // 查它有哪些字段
  for (Field field : clazz.getDeclaredFields()) {

  // 字段名：password，类型：class java.lang.String

  // 查它有哪些方法
  for (Method method : clazz.getDeclaredMethods()) {

  }
  // 输出：
  // 方法：getId
  // 方法：setId
  // 方法：getNickname
  // 方法：setNickname

  反射 = 给类照 X 光。不需要提前知道类的结构，运行时直接扫描它所有的字段、方法、构造器，还能调用私有方法。

---

  反射怎么用的？三个核心能力

1. 获取 Class 对象（三种方式）

  Classc1 = DpUser.class; // 直接通过类名 Class c2 = user.getClass();                   // 通过对象的 getClass
  Class<?> c3 = Class.forName("com.example.mgdemoplus.common.entity.DpUser"); // 通过全限定名字符串
  // 三种方式拿到的是同一个 Class 对象

1. 暴力访问私有字段

  DpUser user = new DpUser();
  Field field = DpUser.class.getDeclaredField("password");  // password 是 private 的
  field.setAccessible(true);   // 暴力破解："别管 private 了，让我进去"
  field.set(user, "123456");    // 直接修改私有字段的值

1. 动态调用方法

  DpUser user = new DpUser();
  Method method = DpUser.class.getDeclaredMethod("setNickname", String.class);
  method.invoke(user, "张三");   // 等价于 user.setNickname("张三")

  反射和注解的关系：现在串起来了

  // Spring 启动后，扫到了你的 Controller：
  @RestController
  @RequestMapping("/dp")
  public class DpFriendMailboxController {
      @Autowired
      private DpFriendSocialService service;  // 这个字段是 private 的！
  }

  // Spring 内部通过反射做了这些事（简化版）：
  Class<?> clazz = DpFriendMailboxController.class;

  // 1. 读类上的注解
  if (clazz.isAnnotationPresent(RestController.class)) {
      // 注册为 Controller
  }

  String path = clazz.getAnnotation(RequestMapping.class).value();  // → "/dp"

  // 2. 遍历字段，找被 @Autowired 标记的
  for (Field field : clazz.getDeclaredFields()) {
      if (field.isAnnotationPresent(Autowired.class)) {
          // 3. 暴力访问 private 字段
          field.setAccessible(true);
          // 4. 去 Spring 容器找 DpFriendSocialService 的对象
          Object serviceBean = springContainer.get(field.getType());
          // 5. 塞进去
          field.set(controllerInstance, serviceBean);
      }
  }

  注解是标签，反射是读标签的手。 Spring 之所以能让 @Autowired
  自动注入，就是启动时用反射把你所有的类扫一遍，看到标签就干活。

  反射的代价

  // 正常调用：快
  user.setNickname("张三");        // 编译时已确定，JVM 直接执行

  // 反射调用：慢
  method.invoke(user, "张三");     // 运行时动态解析，多好几层安全检查

  反射比直接调用慢，但它提供了灵活性——框架不需要提前知道你的类，照样能操作。这就是 Spring、MyBatis 能工作的基础。

  一句话总结

  反射 = 运行时窥探和操作类的内部结构。注解贴标签，反射读标签，框架干活。 @Autowired 的魔法不是魔法，是反射在背后给你
  new 好对象、塞进字段。

  一句话先说结论

  线程池就是一个「管工」的机制——它决定：招多少人、最多招多少人、闲着的人留多久、活多了放哪儿、满了怎么办。

  ---
  用银行柜台来类比

  想象你去银行办业务：

| 参数                                | 类比           | 解释                                        |
| :----------------------------------- | :------------- | :------------------------------------------ |
| corePoolSize 核心线程数              | 常驻柜员       | 平时银行每天都开3个窗口，不管有没有人，这3个柜员都在 |
| maximumPoolSize 最大线程数           | 最多能开的窗口数 | 最多能临时加到8个窗口，不能超过           |
| keepAliveTime 空闲存活时间           | 临时柜员能闲多久 | 临时柜员10分钟没活干，就走人              |
| TimeUnit 时间单位                    | 时间单位       | 10是啥？秒？分钟？得说清楚               |
| workQueue 工作队列                   | 等候区座位     | 5个座位，满了才加开窗口                   |
| threadFactory 线程工厂               | 招人的HR       | 谁负责招柜员，叫啥名字                   |
| rejectedExecutionHandler 拒绝策略    | 大厅满了咋办   | 座位没了、窗口全开满了，再来人怎么办      |

  ---
  一个一个说清楚

  1. corePoolSize（核心线程数）

  线程池里一直活着的线程数，哪怕没事干也不会销毁。

  // 哪怕没任务，这3个线程也活着等你
  ExecutorService pool = new ThreadPoolExecutor(3, ...);

  2. maximumPoolSize（最大线程数）

  线程池能塞多少线程的上限。核心线程 + 临时线程不能超过这个数。

  3. keepAliveTime + TimeUnit（空闲存活时间）

  只对临时线程生效。核心线程不需要这个，它们永远不死。

  临时线程闲了超过这个时间 → 回收销毁。

  4. workQueue（工作队列）

  任务先给核心线程，核心线程忙不过来了 → 进队列排队。

  常见的三种队列：

  - 有界队列 ArrayBlockingQueue：固定座位数
  - 无界队列 LinkedBlockingQueue：无限排（危险，可能OOM）
  - 同步队列 SynchronousQueue：不排队，直接找人干

  5. threadFactory（线程工厂）

  给线程起名的，出问题好排查。

  new ThreadFactoryBuilder().setNameFormat("llm-pool-%d").build()

  6. rejectedExecutionHandler（拒绝策略）

  核心满了、队列满了、max也满了 → 新任务怎么办？四种策略：

  ┌─────────────────────┬──────────────────────────────┐
  │        策略         │             行为             │
  ├─────────────────────┼──────────────────────────────┤
  │ AbortPolicy（默认） │ 直接抛异常                   │
  ├─────────────────────┼──────────────────────────────┤
  │ CallerRunsPolicy    │ 谁提交的谁自己跑（主线程跑） │
  ├─────────────────────┼──────────────────────────────┤
  │ DiscardPolicy       │ 直接扔掉新任务，不吭声       │
  ├─────────────────────┼──────────────────────────────┤
  │ DiscardOldestPolicy │ 扔掉队列里最老的，接新的     │
  └─────────────────────┴──────────────────────────────┘

  面试高频流程题

  "线程池任务提交流程是什么？"

  记住这条链路：

  来一个任务
    → 核心线程有空？→ 是 → 核心线程执行
    → 否 → 队列有空？→ 是 → 扔进队列
    → 否 → 线程数 < max？→ 是 → 创建临时线程执行
    → 否 → 执行拒绝策略

  注意坑点：是先排队再创建临时线程，不是先扩线程再排队！很多人背反了。

  总结口诀

  "核→队→扩→拒"
  一句话：线程池就是「常驻工+临时工+候客区+拒客规则」——核心数、最大数、队列、拒绝策略，这四个是面试重点中的重点。
