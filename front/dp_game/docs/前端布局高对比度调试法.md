# 前端布局高对比度调试法（轮廓探针）

本文记录一种在排查 **CSS / Flex / 裁剪（overflow）** 类问题时非常省时的做法：**用夸张、高识别度的视觉标记「画出边界」**，先回答「改的文件到底有没有生效」「问题出在盒子链的哪一环」，再谈细调数值。

---

## 这种方法通常叫什么

没有唯一「学名」，在不同语境里会听到类似说法，可以混用：

| 说法 | 说明 |
|------|------|
| **轮廓调试 / Outline 调试** | 给可疑容器加 `outline`、`box-shadow`，一眼看见盒子的几何边界。 |
| **高对比度可视化探针** | 故意用洋红、荧光绿等「不可能错过」的颜色，当作探测信号。 |
| **失效假设验证（快速版）** | 先假设「某段样式已加载、某选择器已命中」，用极强样式去 **证伪**：若完全不变，则假设不成立（路径、缓存、构建产物错了）。 |
| **二分排除**（广义） | 与「注释掉一半代码」同属一类思路：用极大/极小的改动快速缩小嫌疑范围。 |

英文社区里也常说 **visual debugging**、**debug outlines**、**"use an ugly color so you can't miss it"**。

---

## 为什么高效

1. **区分两类失败**：「样式没生效」vs「生效了但数值不够」——前者查构建、缓存、选择器特异性；后者才调 `padding`、`z-index`、`overflow`。  
2. **边界问题一目了然**：裁剪、子元素溢出父级、flex 子项被挤扁，往往 **画出父盒子** 就能看出裁切发生在哪一层。  
3. **比只看 DevTools 更快上手**：对不熟悉盒模型的新人，彩色框比 Computed 一长串数字更直观。

---

## 推荐做法（简短流程）

1. **选一个最小可疑容器**（例如 `.dp-game-table__layout`、`.dp-game-table-fit`、`<main>`）。  
2. **加临时样式**（务必标 `[TEMP DEBUG]`，便于全文搜索删除）：  
   - `outline: 10px solid magenta;`  
   - 或 `box-shadow: inset 0 0 0 8px lime;`  
   - 可同时把相关 **CSS 变量** 调成离谱的大值（如 `padding-top: 200px`）。  
3. **刷新页面**：  
   - **完全看不见变化** → 优先怀疑：**没打到这份 CSS**（未构建、JAR/静态目录仍是旧的、URL 指错环境、强缓存）。  
   - **能看见框但问题照旧** → 说明命中了这层，继续 **向内层或外层** 换容器加框，或查 `overflow` / `transform` / `z-index`。  
4. **确认原因后**：删掉所有调试样式与离谱数值，再改成 **合理的 token / clamp**，并补一行注释说明「为何需要这块留白或余量」。

---

## 注意事项

- **调试样式不得合并进主线长期保留**，避免污染主题与无障碍对比度。  
- `outline` **不占布局流**，适合只看边界；若要验证「占位是否挤开布局」，改用 **`padding` / `margin` / `min-height`**。  
- 结合 **`transform: scale` + `overflow: hidden`** 时，轮廓能帮助区分：**究竟是裁剪框太短**，还是 **子元素画出父盒但被上层盖住**（后者还要看 **叠层顺序 `z-index`**）。

---

## 在本项目中的实例

对局页曾用 **洋红外轮廓 + 绿色 inset** 包住 `.dp-game-table__layout`，并临时拉大 `--dp-table-layout-pad-top` 等变量，用于验证：

- 静态资源是否更新；  
- 上牌被裁/被挡究竟发生在 **layout 内** 还是 **顶栏叠层**。

验证完成后应删除临时块，改为正式的 **`--dp-game-table-fit-pad-top`** 等与 mixin 测量一致的变量（见 `dp-game-shell.css`、`dpGameTableFitMixin.js`）。
